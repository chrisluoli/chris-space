RAG 知识问答全链路 · 准确性影响因素分解

四个角色各管什么？—— MCP 知识分片召回工具 · 大模型 · 业务 Skill · 知识库
🛠 智能体平台提供
MCP 知识分片召回工具 — 知识切片、Embedding、检索召回
大模型 — 语义理解、推理、生成回答（仅可使用 S3 本地部署模型，与 Claude 等顶级模型存在能力差距）
💼 业务提供
业务 Skill — 角色设定、规则、输出规范
知识库 — 知识语料内容、覆盖度、结构质量（平台已支持连接 HIHR、8008 问答对、公文系统三大内容源）
👤 用户提问
①

问题理解

Query Understanding
🧠 大模型 📦 业务 Skill

影响因素

💬
用户表述模糊 大模型
口语化、缺少上下文，同一问题可能有多种理解方向，依赖大模型的语义理解能力
🎯
意图识别准确性 大模型 Skill
大模型做通用理解，Skill 定义业务触发场景和关键词表帮助精准路由
🔄
多轮对话上下文 大模型
能否结合历史对话理解当前问题的完整语义
📋
信息采集与追问 Skill
Skill 定义"先问清楚什么再回答"的采集规则，避免在信息不足时猜测作答
🏷️
业务术语映射 Skill
Skill 中定义口语化表述到专业概念的映射，降低理解偏差

常见错误表现

答非所问 — 理解偏了方向，回答不是用户想要的
回答了用户没问的问题 — 过度发散
每轮对话"失忆" — 无法关联上下文
信息不足就猜测作答 — 缺少追问机制导致答错

⚡ 业务 Skill 能带来的提升

在 Skill 中定义触发场景和关键词表，帮助大模型快速识别业务意图并路由到正确处理流程
设计首轮信息采集表（明确"先问清楚什么再回答"），避免在信息不足时猜测作答
定义业务专属术语映射，将口语化表述对应到专业概念，降低理解偏差
②

检索召回

Retrieval
🔧 MCP 召回工具 📚 知识库 📦 业务 Skill

影响因素

✂️
知识切片策略 (Chunking) MCP
文档按什么粒度切割？跨 chunk 的完整知识可能只召回一半
📏
切片大小 MCP
太大 → 噪声多；太小 → 信息不完整
🧬
Embedding 模型质量 MCP
向量化效果决定语义匹配精度，通用模型可能不理解业务术语
🔗
语义 vs 关键词检索策略 MCP
纯语义检索漏精确匹配；纯关键词检索漏语义相关内容
🔢
Top-K / 相关性阈值 MCP
K 太小漏答案，K 太大引入噪声；阈值设置影响召回质量
📚
知识库覆盖度 知识库
知识库里有没有这个问题的答案？没有 = 再好的检索也找不到
📐
知识写作质量 知识库
知识描述是否清晰、结构化？模糊的知识 → 模糊的召回
🏷️
知识组织结构 知识库
是否按主题/场景分章节？有无 metadata 标注？影响召回精准度
🗂️
意图-知识映射规则 Skill
是否定义了"什么问题该查哪段知识"的路由规则？决定召回确定性

常见错误表现

知识库里压根没有答案 → 无米之炊
找到了不相关的内容 → 噪声污染
找到的知识不完整 → 只答了一半
业务术语无法匹配 → 语义理解断层

⚡ 知识库 + 业务 Skill 能带来的提升

知识库 补全知识覆盖度 — 确保业务核心场景都有对应的知识语料，消除"无米之炊"
知识库 结构化知识写作 — 按主题/场景分章节、加标题标注，让 MCP 工具更容易找到正确内容
Skill 定义匹配规则 — 通过意图→章节映射表实现确定性路由，不完全依赖向量相似度
③

知识匹配 / 重排

Reranking & Filtering
🔧 MCP 召回工具 📚 知识库 📦 业务 Skill

影响因素

🏆
Reranker 模型能力 MCP
二次排序精度，排在前面的是否真的是最相关内容
📊
相关性阈值 MCP
低相关度内容混入会误导大模型
⚔️
知识冲突处理 Skill
多条知识矛盾时该信谁？需要 Skill 定义优先级规则
🕐
时效性判断 知识库
知识有更新时需识别哪条是最新有效版本，依赖知识库的版本管理
🔬
知识粒度匹配 知识库
问具体操作细节，召回的却是宏观概述 → 答不到点上，取决于知识库的写作粒度

常见错误表现

用了过时的知识回答 — 新政策/规则没有被优先采用
多条知识堆砌没有优先级 — 大模型不知道该信哪个
回答内容自相矛盾 — 引用了互相冲突的知识源
回答过于宏观笼统 — 没有匹配到细粒度的操作指南

⚡ 知识库 + 业务 Skill 能带来的提升

Skill 定义知识优先级 — 明确多份知识源冲突时的取舍规则（如"操作指南 > 一般参考"）
知识库 标注知识版本和时效 — 让重排时能识别哪些是最新有效内容
知识库 控制知识粒度 — 按操作级别组织知识，避免"问具体流程，只给宏观描述"
④

上下文组装

Context Assembly / Prompt Engineering
📦 业务 Skill 主导

影响因素

🎭
角色设定 (System Prompt) Skill
定义专家身份 vs 无角色设定，专业度和回答风格差异巨大
📝
Prompt 模板设计 Skill
是否明确引导了"先做什么分析，再给什么建议，最后输出什么格式"
📦
上下文窗口限制 大模型
知识太多塞不下需截断，关键信息可能被截掉
📑
知识注入顺序 Skill
大模型对开头和结尾关注度更高 (Lost in the Middle)，重要知识应放前面
📐
输出格式约束 Skill
是否约定了回答结构（表格/分步/Checklist），影响可读性
🚫
行为边界 / 禁止事项 Skill
是否定义了"不能做什么"（如不能编造数据、不能给超范围建议）

常见错误表现

回答没有条理 — 东拉西扯，用户看不出重点
回答风格不稳定 — 有时像专家、有时像闲聊
关键信息被淹没 — 重要提醒埋在长篇回答中间
给了超出业务边界的建议 — 没有约束行为范围
不同用户得到格式差异很大的回答 — 缺少统一模板

⚡ 业务 Skill 能带来的提升（此环节 Skill 主导）

定义专家角色 — 告诉大模型"你是什么领域的专家"，整体提升回答的专业度和一致性
设计输出模板 — 按问题类型定义回答结构（如流程类用步骤表、对比类用表格），保证输出规范
设定禁止事项 — 明确"不能做什么"，防止大模型给出超范围或错误建议
嵌入业务规则 — 将关键业务规则写入 Prompt，大模型生成时作为硬约束自动检查
⑤

模型生成

LLM Generation
🧠 大模型

影响因素

🧠
模型基础能力
不同大模型的推理、理解、指令遵循能力不同，复杂场景差异明显
👻
幻觉 (Hallucination)
编造不存在的数据/事实，听起来很真但实际没有依据
🎯
忠实度 (Faithfulness)
是否忠于检索到的知识生成回答，还是凭"预训练印象"自由发挥
⚡
推理能力
能否综合多条知识做逻辑推理，而非简单拼接复述
🌡️
Temperature / 采样参数
影响回答的确定性和一致性，专业场景应低温避免"创造性发挥"

常见错误表现

编造数据 / 事实 — 幻觉是大模型最典型的问题
忽略检索到的知识 — 用自己的"常识"回答，与知识库矛盾
复杂场景推理错误 — 多条件组合时逻辑链断裂
同一问题每次回答差异大 — 缺少确定性
🧠
此环节由大模型自身能力决定
业务 Skill 和知识库不直接控制大模型的推理能力和幻觉倾向
但好的 Skill（④ 上下文组装）能间接降低此环节的出错率
⑥

输出呈现

Output Delivery
📦 业务 Skill 主导

影响因素

📋
输出结构化 Skill
分点、标题、表格呈现 vs 一大段文字，用户体验差异巨大
🔖
知识溯源 (Citation) Skill
标注信息来源，让用户可以验证，增强信任度
🚀
可操作性 Skill
给出具体可执行的下一步，而非笼统建议
📎
附件 / 模板输出 Skill
能直接输出可用的文件模板（如 Excel），比纯文字描述实用得多

常见错误表现

信息正确但用户看不懂 / 不想看 — 呈现形式差
没有给出具体行动建议 — 只有分析没有结论
无法追溯来源 — 用户不敢采信
需要表格/模板但只给了文字描述 — 不可操作

⚡ 业务 Skill 能带来的提升（此环节 Skill 主导）

定义输出格式规范 — 表格优先、分步骤呈现、关键信息高亮，统一所有回答的呈现质量
要求知识溯源 — 在 Skill 中写明"回答时必须标注来源"，增强可信度
预置操作模板 — 在 Skill assets 中放入 Excel/文档模板，用户需要时直接输出可用文件
✅ 用户看到答案

💡 核心结论
RAG 链路由四个角色协同
🔧
MCP 知识分片召回工具
将知识切片、向量化、检索召回
找知识的引擎
🧠
大模型
语义理解、推理生成
产出回答的大脑
纯依赖模型，无法人工优化
📦
业务 Skill
匹配规则、角色设定、输出规范
业务逻辑的灵魂
📚
知识库
知识语料、覆盖度、写作质量
回答的原材料
