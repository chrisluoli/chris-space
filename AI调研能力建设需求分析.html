🤖 AI调研能力建设需求分析

智能体平台 × 测温系统 × 业务系统

1 需求点分解

以离职系统为例，当前业务系统面向用户开展调研时，传统方式依赖人工推送问卷、人工整理数据，效率低且覆盖率有限。希望引入 AI 智能体 来完成问卷调研动作，实现：

  • 自动化调研：AI Agent 按预设题目与用户进行多轮对话式调研
  • 灵活配置：业务方可自主配置推送时间、渠道、题目、截止日期等
  • 数据合规：智能体平台不存储或仅加密存储敏感数据，完整数据归属业务方
  • 前端统一：智能体平台提供标准对话 SDK，业务方嵌入即可使用
  • 自主规划 +Skill：通过接口注入动态参数

4 数据存储方案

4.1 智能体平台数据处理策略

方案 描述 优劣势 参考案例
方案A：零存储
（推荐）		 ① 题目配置通过 API 实时下发，仅驻留内存
② 对话过程中实时通过 Webhook/回调 将每轮对话回传测温系统
③ 会话结束后，智能体平台清除所有会话内存
④ 不写入任何磁盘/数据库 		数据合规性最强
对网络稳定性要求高
需要设计重试机制保障数据不丢 		无
方案B：加密暂存 ① 对话过程中以 AES-256 加密 暂存到智能体平台
② 密钥由测温系统管理（智能体平台不持有明文密钥）
③ 会话结束后批量回传测温系统，确认接收后立即删除
④ 设置 TTL 兜底（如24h自动清除） 		数据不丢失风险低
合规性略低于方案A
适合网络不稳定场景 		敏感知识、AI问数，加密不存，通过密钥获取用于加载对话页面的聊天记录；如果是访谈调研场景，不需要渲染历史聊天记录，可以考虑直接不存的方案

4.2 数据流转全链路

完整链路（推荐方案A）：
业务系统「推送名单」→ 测温系统「配置场景 + 下发参数」→ 智能体平台「模板+参数拼装」→ 用户「SDK对话」→ 智能体平台「实时回传」→ 测温系统「持久存储」→ 智能体平台「清除内存」→ 测温系统「数据分析」→ 业务系统「业务消费」

5 对话页面提供方案

5.1 方案说明

智能体平台提供可嵌入的对话 JS插件供业务系统引入，业务方零开发成本：

• 支持主题色、Logo、引导语定制
• 内置流式输出、追问、打字机效果
• 业务方一行代码嵌入，平台统一升级自动受益

5.2 数据流

SDK 只是前端渲染层，不改变数据存储策略。
数据流仍然是：用户 → [SDK 前端] → 智能体平台（内存处理）→ 实时回传 → 测温系统（持久化）→ 业务系统（消费报告）。
智能体平台仍然是无状态执行引擎，SDK 前端直连智能体 API，对话数据实时旁路回传测温系统。

6 Skill 工程

6.1 需求

🧠 智能体平台 — 「结构化调研」Skill 模板

  • 固定部分（平台维护）：
  • System Prompt 框架 · 多轮对话管理策略 · 追问引擎（何时追问、几轮、方式）
  • 情感识别 + 安全护栏 · 回答质量评估 · 超时/异常处理
  • 动态部分（每次会话由外部注入）：
  • 场景描述（离职调研 / 满意度调研 / …） · 题目列表 + 分支逻辑
  • 语气风格（温和 / 正式 / 轻松） · 开场白 / 结束语 · 业务自定义规则

📋 测温系统 — 场景配置（业务人员操作）

  • 题目编排（顺序、分支、必答/选答）
  • 追问规则（什么情况下追问、追问几轮）
  • 跳转逻辑（答了A跳到第3题，答了B跳到第5题）
  • 场景描述片段（开场白、结束语、语气风格）
  • 结束条件（全部答完 / 超时 / 用户主动退出）
  • 配置完成后，按智能体平台的 Skill 协议标准打包下发

🎓 业务系统 — 不与 AI 能力侧直接交互

  • 只负责提供业务上下文（用户部门、岗位等），影响调研策略的参数
  • 这些参数随名单一起传给测温系统，由测温系统注入场景配置中

6.2 新场景扩展方式

每创建一个新的调研场景（如满意度调研），流程是：

① 测温系统新建一个场景模板，配好题目、规则、风格等「业务参数」
② 创建会话时按 Skill 协议打包下发给智能体平台
③ 智能体平台用同一个「结构化调研」Skill 模板执行
④ 智能体平台侧不需要任何新增配置——不同场景只是注入的参数不同
